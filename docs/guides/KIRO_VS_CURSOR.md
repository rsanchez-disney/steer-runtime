# Kiro vs Cursor — Technical Comparison

How steer-runtime workspaces map to each runtime at every configuration level.

## Architecture overview

```
                    steer-runtime (source of truth)
                    ┌──────────────────────────────┐
                    │  workspaces/                  │
                    │  profiles/                    │
                    │  shared/steering/             │
                    │  shared/tools/mcp-servers/    │
                    └──────────────┬───────────────┘
                                   │
                         koda sync / workspace apply
                                   │
                    ┌──────────────┴───────────────┐
                    │                              │
              ┌─────▼─────┐                 ┌─────▼─────┐
              │   Kiro     │                 │  Cursor   │
              │  ~/.kiro/  │                 │ ~/.cursor/│
              └───────────┘                 └───────────┘
```

## Configuration levels

### Level 1: Global (user-wide)

| Aspect | Kiro | Cursor |
|--------|------|--------|
| Location | `~/.kiro/` | `~/.cursor/` |
| MCP servers | `~/.kiro/settings/mcp.json` | `~/.cursor/mcp.json` |
| Agents | `~/.kiro/agents/*.json` + `prompts/*.md` | `~/.cursor/agents/*.md` |
| Steering (always) | `~/.kiro/steering/*.md` | User Rules (Settings UI) |
| Skills | `~/.kiro/skills/*/SKILL.md` | `~/.cursor/skills/*/SKILL.md` |
| Context | `~/.kiro/context/*.md` | Not supported globally |
| Settings | `~/.kiro/settings/kite.json` | Cursor Settings UI |
| When synced | `koda workspace apply` | `koda workspace apply` (auto) |

### Level 2: Workspace (team-scoped)

| Aspect | Kiro | Cursor |
|--------|------|--------|
| Location | `~/.kiro/workspaces/<name>/` | No equivalent (uses global + project) |
| Purpose | Isolated per-team context | N/A — Cursor doesn't have workspace isolation |
| Agents | Scoped to workspace session | Agents go to `~/.cursor/agents/` (global) |
| MCP | Copy of global with overrides | Same as global |
| Activation | `koda chat --ws <name>` (sets KIRO_HOME) | N/A |

### Level 3: Project (repo-scoped)

| Aspect | Kiro | Cursor |
|--------|------|--------|
| Location | Project root (`.kiro/` or via KIRO_HOME) | `<project>/.cursor/` |
| Rules/Steering | Steering files in KIRO_HOME | `.cursor/rules/*.mdc` |
| Skills | In KIRO_HOME skills/ | `.cursor/skills/*/SKILL.md` |
| MCP | Via KIRO_HOME settings/ | `.cursor/mcp.json` (overrides global) |
| Agents | Via KIRO_HOME agents/ | `.cursor/agents/*.md` (overrides global) |
| Context | Via KIRO_HOME context/ | Rules with `alwaysApply: true` |
| Committable | No (KIRO_HOME is ephemeral) | Yes (`.cursor/` lives in repo) |
| Generated by | `koda chat --ws` (sets env) | `koda cursor --ws --dir <path>` |

## File format differences

### Rules/Steering frontmatter

**Kiro** (`.md`):
```yaml
---
inclusion: always
description: REST API conventions
---
```

```yaml
---
inclusion: fileMatch
fileMatchPattern: ["**/*Controller.java"]
description: REST API conventions
---
```

**Cursor** (`.mdc`):
```yaml
---
alwaysApply: true
description: "REST API conventions"
---
```

```yaml
---
alwaysApply: false
globs: ["**/*Controller.java"]
description: "REST API conventions"
---
```

### Translation table

| Kiro frontmatter | Cursor frontmatter |
|-----------------|-------------------|
| `inclusion: always` | `alwaysApply: true` |
| `inclusion: auto` + `description` | `alwaysApply: false` + `description` |
| `inclusion: fileMatch` + `fileMatchPattern` | `alwaysApply: false` + `globs` |

### Agent format

**Kiro** — two files:
```json
// agents/backend.json
{
  "name": "backend",
  "description": "Java/Spring specialist",
  "prompt": "backend.md",
  "tools": ["code", "fs_read", "fs_write", "@jira/*"],
  "allowedTools": ["@jira/*", "@confluence/*"]
}
```
```markdown
<!-- prompts/backend.md -->
## Identity
You are a backend specialist...
```

**Cursor** — single file:
```markdown
<!-- .cursor/agents/backend.md -->
---
description: "Java/Spring specialist"
alwaysApply: false
---

## Identity
You are a backend specialist...
```

### Skills format

**Identical** — both use `SKILL.md` with the same frontmatter:
```yaml
---
name: create-rest-api
description: Create a REST API endpoint following hexagonal architecture
---

# Create REST API Endpoint
...
```

### MCP format

**Nearly identical** — both use `{"mcpServers": {...}}`:

```json
{
  "mcpServers": {
    "jira-cloud": {
      "command": "node",
      "args": ["path/to/bundle.cjs"],
      "env": { "JIRA_URL": "https://..." }
    }
  }
}
```

Cursor additionally supports:
- `${env:VAR}` interpolation (Kiro uses raw values)
- `envFile` field for `.env` loading
- `url` + `headers` for remote SSE/HTTP servers (same as our Compass)

## Behavioral differences

| Behavior | Kiro | Cursor |
|----------|------|--------|
| Session persistence | ACP session (in-memory until compact) | None (relies on PROJECT_MEMORY.md rule) |
| Agent switching | `/agent <name>` or subprocess respawn | `@agent-name` mention in chat |
| Delegation | Orchestrator → subagent tool call | Agent-selected rules + subagent `.md` files |
| Tool approval | Trust levels (all/supervised/none) | Allow once/always, reject |
| Context window | Single window, compact when full | Per-session, no persistence |
| Headless/CI | `kiro-cli --print` via `koda ci run` | `agent -p --force` |
| Planning mode | `koda plan` (kiro_planner agent) | `/plan` or `agent --mode=plan` |

## When to use which

| Scenario | Recommended |
|----------|-------------|
| Terminal-first workflow | Kiro (`koda chat --ws`) |
| IDE-integrated coding | Cursor (`koda cursor --ws`) |
| CI/CD automation | Either (`koda ci run` or `agent -p`) |
| Multi-agent orchestration | Kiro (stronger delegation protocol) |
| File-scoped rules (auto-attach by glob) | Cursor (native `.mdc` glob support) |
| Team sharing via git | Cursor (`.cursor/` is committable) |
| Centralized profiles | Kiro (workspace materialization) |

## Koda commands by runtime

| Command | Kiro | Cursor |
|---------|------|--------|
| `koda chat` | ✅ Launches kiro-cli | ✅ Auto-detects `.cursor/` → launches `agent` CLI |
| `koda chat --ws` | ✅ kiro-cli with workspace | ✅ `--target cursor` or auto-detect |
| `koda cursor --ws` | N/A | ✅ Generates project `.cursor/` |
| `koda workspace apply` | ✅ Materializes to `~/.kiro/` | ✅ Auto-syncs `~/.cursor/` (MCP + agents) |
| `koda ci run` | ✅ `kiro-cli --print` | ❌ (use `agent -p` directly) |
| `koda plan` | ✅ kiro_planner agent | ❌ (use `agent --mode=plan` directly) |
| `koda setup runtime` | Select kiro/cursor/both | — |
| `koda doctor` | ✅ Kiro health checks | ✅ Cursor CLI + MCP + agents checks |

## Auto-detection

When you `cd` into a project directory:

| Directory contains | `koda chat` behavior |
|-------------------|---------------------|
| Nothing | Uses Kiro (default) |
| `.cursor/` | Uses Cursor agent CLI |
| `.cursor/.koda-meta.json` | Uses Cursor + infers workspace + default agent |

Explicit `--target kiro` or `--target cursor` always overrides auto-detection.
