# Workspace Technical Reference

Complete specification for creating, configuring, and maintaining team workspaces in steer-runtime.

---

## Overview

A workspace is a self-contained configuration bundle that gives a team's AI agents the right context, tools, rules, and knowledge. When applied, it materializes to `~/.kiro/workspaces/{name}/` where kiro-cli reads agents, prompts, context, and hooks.

---

## Naming Convention

Every workspace must include a **tier suffix** to indicate its scope:

| Tier | Pattern | Purpose | Example |
|------|---------|---------|---------|
| Vertical | `{name}-vertical` | Leadership oversight over multiple teams | `payments-vertical` |
| Team | `{name}-team` | Dev/ops team workspace | `shield-team`, `retail-restaurant-team` |
| Sustainment | `sustainment-{studio}` | Ops-scoped to a managed services catalog studio | `sustainment-kaos` |
| Project | `app-{name}` | Single project/app scope within a team | `app-config-studio` |

### Rules

- **kebab-case** — all lowercase, hyphens only
- **Directory = name** — directory name must match the `name` field in workspace.json
- **No redundant words** — don't include "workspace" or "studio-"
- **Short and recognizable** — prefer acronyms over full descriptions

## Directory Structure

```
workspaces/{name}/
├── workspace.json              ← Configuration (required)
├── README.md                   ← Setup instructions (recommended)
├── context/                    ← Workspace-specific context files
│   ├── team_context.md         ← Team roster, domains, workstreams
│   ├── service_repo_mapping.md ← Repo → service → host → branch map
│   ├── splunk_services.md      ← Splunk indexes and query patterns
│   ├── jira_query_context.md   ← JQL patterns and components
│   ├── build_commands.md       ← Per-repo build/test commands
│   ├── descriptions/           ← Agent role descriptions (for delegation)
│   └── _dynamic/               ← Auto-generated by hooks (do NOT edit manually)
├── rules/                      ← Workspace-specific rules (beyond common/)
├── skills/                     ← Workspace-specific skills (flat .md files)
│   └── {skill-name}.md         ← Skill definition (MUST be flat, not subdirectory)
├── profiles/                   ← Profile overrides (agents, prompts per profile)
│   └── {profile-id}/
│       ├── agents/             ← Agent JSON overrides
│       └── prompts/            ← Prompt overrides
├── common/                     ← Transversal files installed across all profiles
│   ├── skills/                 ← Skills shared across profiles
│   └── context/                ← Context shared across profiles
├── steering/                   ← Steering files (behavioral directives)
├── templates/                  ← Code generation templates
└── tools/                      ← Workspace-level MCP servers
    └── mcp-servers/
        └── {server-name}/
            ├── mcp-meta.json
            └── dist/index.cjs
```

### Child Workspaces (extends)

```
workspaces/{parent-team}/
├── workspace.json              ← Parent workspace
├── context/                    ← Inherited by all children
└── {child-name}/
    ├── workspace.json          ← "extends": "{parent-team}"
    └── context/                ← Child-specific additions
```

## workspace.json Schema

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Unique workspace identifier (must match directory name) |
| `description` | string | Human-readable description |
| `team` | string | Team name for display |
| `profiles` | string[] | Profile IDs to install |
| `default_agent` | string | Agent to use by default in chat |

### Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `extends` | string | Parent workspace name for inheritance |
| `projects` | object[] | Repository entries (see Project Entry below) |
| `rules` | string[] | Rule file names from `common/rules/` |
| `enable_tools` | boolean | Enable thinking, todo, knowledge tools |
| `jira_prefix` | string \| string[] | Jira project prefix(es) (e.g., `"DPAY-"` or `["FNB-", "MERCH-"]`) |
| `jira_host` | string | Jira instance URL (default: `jira.disney.com`) |
| `jira_instance` | string | Named Jira instance (e.g., `"myjira"`) — alternative to jira_host |
| `jira_board_id` | number | Default Jira board ID for sprint queries |
| `jira_board_url` | string | Direct URL to the team's Jira board |
| `jira_components` | string[] | Jira component names for filtering |
| `jira_cloud` | boolean | Whether the Jira instance is Atlassian Cloud |
| `jira_url` | string | Full Jira base URL (legacy, prefer jira_host) |
| `jira_custom_fields` | object | Custom Jira field mappings (e.g., `{"story_points": "customfield_10003"}`) |
| `services` | string[] | Service bank names from `shared/services/` |
| `channels` | string[] | Channel bank names from `channels/` |
| `workspace_path` | string | Base path for project repos (supports env vars) |
| `managed_studios` | string[] | Studio directories from managed-services-catalog |
| `catalog_scope` | string | Alternative to managed_studios — scope filter for catalog |
| `teams` | object[] | Teams in a vertical (leadership workspaces only) |
| `exclusive_overrides` | object | Profile dirs that replace (not merge) global |
| `harness` | object | Harness pipeline configuration |
| `wiki_url` | string | Team wiki/Confluence URL |

### Valid Profile IDs

`dev-core`, `dev-web`, `dev-mobile`, `dev-python`, `dev-infra`, `dev-ui`, `dev-php`, `dev-dotnet`, `dev-ai`, `ba`, `qa`, `ops`, `pm`, `steer-master`, `leadership`, `sustainment`, `core`, `inspector`, `design`, `presales`, `cloudops`

Alias: `dev` expands to `dev-core` + `dev-web` + `dev-mobile` + `dev-python` + `dev-infra`

## Project Entry Schema

Each entry in `projects` describes a repository:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Project display name |
| `path` | string | Directory name (relative to `workspace_path`) |
| `repo` | string | Repository path — `org/repo` format (no host prefix) |
| `host` | string | Git host: `github.disney.com` (default) or `gitlab.disney.com` |
| `memory_bank` | string | Memory bank template name (optional) |
| `description` | string | Human-readable description (optional) |
| `studio` | string | Studio grouping label (optional) |
| `alias` | string | Short alias for CLI references (optional) |

### Host Routing

| `host` value | Clone URL | Agent tools |
|---|---|---|
| `github.disney.com` (or omitted) | `git@github.disney.com:org/repo.git` | `@github/*`, `gh` CLI |
| `gitlab.disney.com` | `git@gitlab.disney.com:org/repo.git` | `@gitlab/*` |

### Example

```json
{
  "name": "my-service",
  "path": "my-service",
  "repo": "MyOrg/my-service",
  "host": "github.disney.com",
  "description": "Backend API service (Java/Spring Boot)"
}
```

## Inheritance

A workspace can `extend` a parent. The child inherits then adds/overrides:

| Field | Merge Strategy |
|-------|---------------|
| `profiles` | Additive (union) |
| `rules` | Additive (union) |
| `services`, `channels` | Additive (union) |
| `context/` files | All copied (root-first, child wins) |
| `description`, `team`, `default_agent`, `jira_prefix` | Child overrides parent |
| `projects` | Child only (not inherited) |
| `workspace_path` | Child overrides if set |

Chains can go multiple levels deep. Resolution walks bottom-up with cycle detection.

## Teams Entry Schema (Verticals Only)

Used in `{name}-vertical` workspaces to declare the teams they oversee:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | Team display name |
| `workspace` | string | Workspace name for this team |
| `jira_projects` | string[] | Jira project keys this team owns |
| `board_ids` | number[] | Jira board IDs for sprint queries |
| `studio` | string | Managed services catalog studio |
| `studio_id` | number | Studio numeric ID |
| `team_id` | number | Team numeric ID (for API queries) |
| `dashboard_id` | number | Dashboard ID for metrics |
| `notes` | string | Free-text notes about the team |

## workspace_path

Supports environment variable expansion for cross-platform compatibility:

| Format | Example | Platform |
|--------|---------|----------|
| `~` | `~/Workspace/Disney` | All |
| `${VAR}` | `${WORKSPACE_ROOT}/Disney` | All (recommended) |
| `$VAR` | `$HOME/Workspace` | macOS/Linux |
| `%VAR%` | `%USERPROFILE%\Workspace` | Windows |

Path separators are normalized automatically — always use `/` in JSON.

## Context Files

Recommended context files for agent enrichment:

| File | Purpose | Used by |
|------|---------|---------|
| `team_context.md` | Team name, members, Jira project, domains | All agents |
| `service_repo_mapping.md` | Service → repo → language → deploy | All agents |
| `splunk_services.md` | Splunk indexes, query patterns, dashboards | `splunk_query_agent` |
| `jira_query_context.md` | Custom fields, JQL patterns, components | `story_analyzer_agent` |
| `build_commands.md` | Per-repo build/test/run commands | `devops_runner_agent` |
| `governance.md` | Review processes, coding standards, SLAs | All agents |

### Dynamic Context (`context/_dynamic/`)

Auto-generated by hooks during workspace apply:

| File | Generated by | Content |
|------|-------------|---------|
| `catalog-index.md` | `catalog-index.sh` | BAPP IDs for managed_studios scope |
| `delegation-map.md` | `agent-registry.sh` | Available agents per profile |
| `agent-registry-full.md` | `agent-registry.sh` | Complete agent descriptions |

## Skills

Skills are flat `.md` files with frontmatter placed directly in `skills/`:

```markdown
---
name: my-skill-name
description: When to invoke this skill
disable-model-invocation: true
---

# Skill Title

## Step 1 — ...
## Step 2 — ...
```

**Important:** Skills must be flat files (`skills/my-skill.md`), NOT subdirectories. kiro-cli discovers `*.md` files directly in the skills directory.

## Materialization

When you run `koda workspace apply {name}` or `koda chat --ws {name}`:

1. **Resolves inheritance** — walks `extends` chain, merges profiles/rules
2. **Installs profiles** — global profiles + workspace profile overrides
3. **Copies rules** — from `common/rules/` for declared rule names
4. **Copies workspace dirs** — rules/, context/, templates/, skills/ from workspace chain
5. **Installs steering** — steering files from workspace chain
6. **Installs MCP bundles** — workspace-level MCP servers
7. **Injects tokens** — applies configured PATs to agent JSON configs
8. **Runs hooks** — agentSpawn hooks generate `_dynamic/` content
9. **Writes snapshot** — `settings/workspace.json` records the resolved state

### Output Directory

Materialized workspaces live at:
```
~/.kiro/workspaces/{name}/
├── agents/         ← Agent JSON configs (with tokens injected)
├── prompts/        ← Agent prompt files
├── context/        ← Context files (including _dynamic/)
├── steering/       ← Behavioral directives
├── rules/          ← Applied rules
├── skills/         ← Discoverable skills
├── hooks/          ← agentSpawn/lifecycle hooks
├── tools/          ← MCP server bundles
├── settings/       ← workspace.json snapshot + mcp.json
└── templates/      ← Code generation templates
```

### How kiro-cli Uses It

- `KIRO_HOME=~/.kiro/workspaces/{name}/` → hooks, agent discovery
- `file://.kiro/*` resources → resolved from `~/.kiro/` (global copy)
- Agent JSON `resources[]` → loaded as system context into the LLM
- agentSpawn hooks → run at session start, output injected into context

## Workspace Lifecycle Commands

```bash
koda workspace list                    # List with hierarchy
koda workspace show {name}             # View details
koda workspace apply {name}            # Materialize + install
koda chat --ws {name}                  # Apply + launch chat
koda sync                              # Pull latest steer-runtime
koda upgrade                           # Update binary + refresh workspaces
```

## Validation

Run workspace validation:
```bash
python3 scripts/validate-workspaces.sh
```

Checks:
- Required fields present
- Name follows naming convention
- Repo format is `org/repo` (no host prefix)
- Profiles are valid IDs
- JSON is parseable

---

## Migration (Renaming)

When renaming a workspace, add to `workspaces/renames.json`:
```json
{
  "old-name": "new-name"
}
```

Koda reads this on sync and auto-migrates users' `activeWorkspace` setting.
