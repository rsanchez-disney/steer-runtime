# Skill: Sync agents to steering files

Use when the user asks to load, sync, or convert agents from `~/.kiro/agents/` into Kiro steering files.

## What this skill does

Reads agent definitions from `~/.kiro/agents/*.json`, finds their associated prompt file in `~/.kiro/prompts/`, and generates:

1. **Per-agent manual steering files** in `~/.kiro/steering/` prefixed with `agent-`
2. **A manual-inclusion agent routing file** at `~/.kiro/steering/70-agent-routing.md` that teaches Kiro IDE how to delegate to any installed agent

## Steps

### Part 1: Per-agent steering files

1. Ensure `~/.kiro/steering/` exists (create it if missing)
2. List all `.json` files in `~/.kiro/agents/`
3. For each agent JSON:
   a. Read the JSON and extract `name`, `description`, and `prompt` fields
   b. Derive the agent key from the filename (e.g. `my_agent.json` → `my_agent`)
   c. Read the corresponding `.md` file from `~/.kiro/prompts/{prompt}`
   d. Generate a steering file at `~/.kiro/steering/agent-{agent_key}.md` with this format:

```markdown
---
inclusion: manual
---
# Agent: {name}

> {description}

{content of the prompt .md file}
```

4. If the agent JSON has `hooks`, append a section documenting them:

```markdown
## Hooks (reference only)

- preToolUse: {description of each hook}
- postToolUse: {description of each hook}
```

5. If the agent JSON has `tools` that reference MCP servers (e.g. `@bruno/*`, `@github/*`, `@jira/*`), append:

```markdown
## MCP tools

This agent expects access to: {list of MCP tool patterns}
```

6. Skip agents whose steering file already exists and has the same content (idempotent).

### Part 2: Agent routing steering file

After processing all agents, generate `~/.kiro/steering/70-agent-routing.md` — a manual-inclusion
steering file that makes every installed agent available to Kiro IDE on demand.

1. Read `~/.kiro/settings/profiles.json` if it exists. Use it to group agents by profile.
   If not found, list all agents ungrouped under a single "Installed Agents" heading.

2. Generate the file with this structure:

```markdown
---
inclusion: manual
description: Agent registry and delegation patterns for invokeSubAgent routing
---

# Agent Routing for Kiro IDE

Kiro IDE does not read `~/.kiro/agents/*.json` directly. This steering file bridges the gap
by teaching you how to delegate tasks to specialist agents via `invokeSubAgent`.

## How to delegate

Use `invokeSubAgent` with `name: "general-task-execution"`. Read the agent's prompt from
`~/.kiro/prompts/{prompt_file}` and pass it as instructions along with the user's request.

## Agent Registry

{for each profile group}
### {profile_id}

| Agent                | Description                    | Prompt File          |
|----------------------|--------------------------------|----------------------|
| `{name}`             | {description}                  | `{prompt}`           |
{end for each}

## Delegation template

When delegating, read `~/.kiro/prompts/{prompt_file}` first, then:

    invokeSubAgent(
      name: "general-task-execution",
      prompt: "<prompt file content>\n\nTask: <user request>"
    )

## Notes

- Always prefer the specialized agent over handling the task yourself when one matches.
- For the full agent hierarchy, see the `AGENTS.md` steering file.
- Generated by the `sync-agents-to-steering` skill. Re-run to update after installing new profiles.
```

3. The routing file must use `inclusion: manual` front-matter so it is not injected into every
   prompt. Instead, the always-on `agent-delegation-trigger.md` rule detects key phrases like
   "use sub agent for", "delegate to", or "route to" and instructs the agent to read
   `70-agent-routing.md` on demand via `readFile`. This keeps the ~6 KB routing table out of
   context until it is actually needed.

4. Overwrite the previous `70-agent-routing.md` on every sync (it is fully generated).

### Part 3: Report

Report:
- How many per-agent steering files were created or updated
- How many were skipped (unchanged)
- Whether the routing file was generated and how many agents it contains

## Usage

User says: "sync my agents" or "load agents as steering" or "convert agents to steering"

Then run this skill by reading `~/.kiro/agents/` and generating the steering files.

## Notes

- Per-agent files use `inclusion: manual` so they are only loaded when the user explicitly
  references them with `#agent-{agent_key}` in chat.
- The routing file (`70-agent-routing.md`) uses `inclusion: manual` so it is not injected into
  every prompt. The companion `agent-delegation-trigger.md` rule (always-on, ~500 bytes) detects
  delegation phrases and instructs the agent to read the routing table on demand. Users can also
  reference it explicitly with `#70-agent-routing` in chat.
- The steering file key is derived from the filename, not the `name` field, to avoid silent
  overwrites when two agents share the same name.
- The original agent JSON hooks cannot be auto-converted to Kiro hooks format. They are
  documented as reference for the user to recreate manually if needed.
- If a prompt `.md` file is not found in `~/.kiro/prompts/`, log a warning and skip that agent
  in the per-agent files, but still include it in the routing table.

  ## Configuration Paths

- Agent definition: `~/.kiro/agents/ui.json`
- Prompt: `~/.kiro/prompts/webui-developer.md`
- Skills: `~/.kiro/skills/opsheet-ui-feature-implementation.md`, `~/.kiro/skills/opsheet-ui-bugfix.md`, `~/.kiro/skills/opsheet-create-pr.md`
- Steering: `~/.kiro/steering/agent-ui.md`
- Config root: `~/.kiro/` (resolves to `$USERPROFILE/.kiro/` or `$HOME/.kiro/`)
