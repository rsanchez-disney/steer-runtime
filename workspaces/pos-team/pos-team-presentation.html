Anatomy of a Great Prompt

How to communicate effectively with AI agents in the POS Team workspace

📝 Context 🎯 Task 🚫 Restrictions 📋 Format ✅ Verification

The 5 Components of an Effective Prompt

📝 1. Context — What the agent needs to know

Provide background information so the agent understands the situation. Include relevant ticket IDs, business models, affected modules, and current state of the code.

✅ "We're working on the ActivateX POS app, specifically the Table Service business model. The receipt printing feature was recently refactored in POS-19892 and we're seeing duplicate items on printed receipts when tips are added in offline mode." ❌ "Fix the printing bug"

🎯 2. Task — What you want done

Be specific about the desired outcome. State the action clearly: implement, analyze, review, fix, generate, plan.

✅ "Implement POS-20029 — fix duplicate redeemed items on printed receipts when a DDP offline transaction has a tip added. The receipt should only show each redeemed item once regardless of tip addition." ✅ "Generate a daily bug report for DSP 2.1.1, 2.1.2, and 2.1.3 releases" ✅ "Review the code changes in the current branch for security and quality"

🚫 3. Restrictions — What NOT to do

Set boundaries and constraints. Tell the agent what to avoid, what not to modify, and what rules to follow.

✅ "Do not modify the existing ReceiptBuilder interface — only extend it. Don't change any behavior for Merchandise or QSR business models. Keep backward compatibility with the current gRPC contract." ✅ "Don't add new dependencies. Use only existing libraries in the project." ✅ "Never write code before I approve the spec."

📋 4. Format — How to present the output

Specify the structure, language, or template you want the response in.

✅ "Present the implementation plan as a table with columns: Task #, Description, Agent, Complexity (S/M/L), Dependencies" ✅ "Generate the PR description using our standard template with Solution Summary, Changes Strategy, and Testing Checklist sections" ✅ "Output the report in markdown with sections matching our daily report format"

✅ 5. Verification — How to confirm success

Define acceptance criteria so the agent knows when the task is complete and correct.

✅ "Verify that: - Unit tests pass with ≥90% coverage on new code - The fix works for all three business models - No existing tests are broken - Gradle assembleDebug compiles successfully" ✅ "The report is correct if all bug counts match the Jira queries and no fabricated data appears in any section."

Complete Prompt Examples for POS Team

📱 Android Implementation

Implement POS-20012. Context: This is a Table Service bug where combined check-printing is not showing auto-apply gratuity on the receipt. The gratuity is correctly applied in the transaction but missing from print output. Restrictions: - Don't modify the TenderProcessor - Keep feature-flagged behind GRATUITY_PRINT_FIX - Must work for both online and offline modes Verification: - Unit test for receipt generation with gratuity - Test both combined and single check scenarios - Gradle build passes

📊 Daily Report Generation

Generate daily DSP reports for all releases. Context: It's July 3, 2026. We need the daily bug status report for DSP 2.1.1, 2.1.2, and 2.1.3. Format: Use the standard daily report template with all 5 sections (Status, Critical, Promotions, New Bugs, Risks). Restrictions: - Paginate all Jira queries completely - Never fabricate data — show "None" if empty - One file per release Verification: Bug counts must match JQL totals.

🏗️ Architecture Decision

Design an ADR for extracting the reduction service from the Connect monolith. Context: The reduction calculation is currently in Connect (PHP) but is needed by both the React frontend and the Android app via gRPC. Current approach creates coupling. Restrictions: - Must maintain backward compatibility - No shared database access - Feature-flagged migration path Format: Use our ADR template with Mermaid diagrams for the system interaction.

✅ QA Coverage Validation

Validate regression coverage for POS-19406 against epic POS-1429. Context: GLB 2.1.3 release QA phase starting. Need to verify test set completeness before execution begins. Format: Coverage matrix with requirement mapping, gap analysis with severity scoring, and recommendations. Verification: Every AC from the epic must map to at least one test case or be flagged as a gap.

Pro Tips

💡 Use Jira Ticket IDs

Always reference the ticket (POS-XXXXX). The agents can fetch full details automatically via Jira MCP.

💡 Specify the Business Model

POS has 3 modes: Merchandise, QSR, Table Service. Tell the agent which ones are affected to avoid blind spots.

💡 Let the Orchestrator Route

You don't need to pick the right agent — just describe your task and the orchestrator will delegate to the specialist.

Triggering Full SDLC Pipelines

🏗️ Backoffice Pipeline (7 Stages)

Say "implement" + a Jira ticket to trigger the full pipeline:

implement POS-19542 build this story: POS-20100 work on CONFE-1234

Pipeline: Analyze → Explore → Plan → 🚦 Gate 1 → Implement → Test → Review → 🚦 Gate 2 → Document

📱 Mobile Pipeline (Spec-First)

Describe the feature or provide a Jira ticket. The architect will present a spec for approval before any code is written.

Implement POS-20012 — fix the gratuity display on combined check receipts for Table Service. Must be feature-flagged.

Pipeline: Understand → Plan → Spec → 🚦 Approve → Implement → Test → Quality Review → Deliver

Tips for Best Results

✅ Do

  • Include Jira ticket IDs (agents auto-fetch details)
  • Specify the business model (Merchandise, QSR, Table Service)
  • State restrictions upfront (don't modify X, keep backward compat)
  • Wait for gate approvals — review the plan before implementation
  • Provide feedback at gates to revise the approach

❌ Don't

  • Don't paste Jira content manually — let agents fetch it
  • Don't skip the spec/plan approval step
  • Don't specify which sub-agent to use — let the orchestrator route
  • Don't give multi-domain requests without clarifying priority
  • Don't use "fix/" or "feat/" branch prefixes (backoffice uses TICKET-ID-description)
